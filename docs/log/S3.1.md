# S3.1 — Projects index with filtering

Status: done · Started: 2026-07-08 · Shipped: 2026-07-09

## Ideate

The index is the first page shaped by the school/individual amendment.
Structure chosen: one `/projects` route with two track sections (School
work, Individual work), category filter chips applying across both.
Separate routes per track were rejected — two thin pages beat one useful
one only after the catalog grows; revisit if it does.

Scope addition, reasoned: authored **parallel-spgemm.mdx** as the second
case study. A one-card index can't demonstrate filtering or grouping, and
SpGEMM was confirmed by Shao's review pass except one optional bracketed
sentence (dropped). Paper attachment stays open as F6.

## Decisions

- Filter state is plain `useState` + `history.replaceState`, with the URL
  param applied in an effect after hydration — deliberately NOT
  `useSearchParams`, which would force a Suspense fallback into the static
  export and strip the cards out of the prerendered HTML. All cards ship
  in the HTML; deep-links (?category=…) apply on hydration.
- Filter chips: real `<button>`s with `aria-pressed`, 44 px touch targets,
  visible focus ring — satisfies the keyboard+ARIA acceptance.
- Empty states: filtered-empty ("no projects in this category yet") vs.
  track-empty ("write-ups in progress" — Individual work until personal
  projects are authored).
- ProjectCard refactored to a normalized `ProjectCardData` view-model;
  landing (interim data) and index (validated frontmatter) both map into
  it. Nav gains its first page link (Projects).
